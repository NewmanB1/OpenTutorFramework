1. Purpose
This specification defines the human authorization, governance, and safety control system for all state-changing operations in OpenTutor.
It ensures that:
No change affecting student learning behavior can occur without validated human approval.
This system is designed to prevent:
uncontrolled pedagogical drift
unsafe automated optimization
unauthorized system modifications
propagation of biased or unvalidated learning policies
2. Core Governance Principle (MUST)
Human Authority Rule
All meaningful changes to educational behavior MUST be explicitly authorized by a human in a valid governance role.
No subsystem (including LLMs, experiment engines, or plugins) may override this rule.
3. Scope of Governance Enforcement
3.1 Operations Requiring Authorization (MUST)
The following actions MUST NOT execute without valid approval:
A. Learning System Changes
pedagogy plugin activation or modification
knowledge bundle updates affecting facts or structure
changes to student state model logic
updates to Bayesian update rules
modification of evaluation metrics
B. Experimentation
creation of experiments
activation of experimental variants
rollout of experimental results into production behavior
C. Runtime System Changes
composition engine rule changes
plugin execution ordering changes
context budgeting changes
determinism rule modifications
D. Safety / Adaptation System Changes
ADAPTATION_POLICY_ENGINE updates
metric weighting changes
reward function modifications
3.2 Operations That MAY Require Authorization
These may be configurable but default to requiring approval:
UI/UX changes affecting learning flow
accessibility mode changes
creative module behavior changes
telemetry schema changes
4. Governance Roles (CANONICAL SOURCE OF TRUTH)
All authorization is tied to explicit roles.
4.1 Role Definitions
Role	Description
Educator	Approves pedagogical changes and learning content behavior
Curriculum Designer	Validates subject and content structure
Research Lead	Approves experiments and learning science changes
Safety Reviewer	Ensures cognitive, emotional, and developmental safety
System Administrator	Final authority for deployment and runtime changes
External Auditor (optional)	Independent review authority for compliance validation
4.2 Role Constraints
Roles are assigned explicitly and cryptographically bound to identity
No role may self-approve changes it originated (Separation of Duties Rule)
Safety Reviewer approval is REQUIRED for any high-risk change
5. Authorization Levels (RISK MODEL)
Each action is classified before approval:
Level	Definition	Required Approvals
LOW	UI, cosmetic, non-learning changes	1 Educator OR Admin
MODERATE	Pedagogy or experiment changes	Educator + Research Lead
HIGH	Learning behavior, metrics, or system logic changes	Educator + Safety Reviewer + Admin
CRITICAL	Global learning policy or system-wide behavioral changes	ALL roles + external audit recommended

7. Cryptographic Verification (MANDATORY)
7.1 Signature Requirements
All approvals MUST be:
cryptographically signed
bound to a specific user identity
bound to a specific action hash
7.2 Verification Rules
Before execution, system MUST verify:
signature validity
identity-role mapping
timestamp validity (no future-dated approvals)
integrity of target action hash
If any check fails:
EXECUTION MUST BE BLOCKED
7.3 Anti-Replay Protection
Each authorization MUST include:
nonce
action hash
expiration timestamp
Reused or replayed approvals are invalid.
8. Authorization Lifecycle
Every governed action follows this deterministic pipeline:
Step 1: Proposal
generated by human or system
Step 2: Risk Classification
assigned by Adaptation Policy Engine or validator
Step 3: Required Role Determination
based on risk level and action type
Step 4: Approval Collection
signatures gathered from required roles
Step 5: Validation
cryptographic + structural verification
Step 6: Activation
system executes action
Step 7: Audit Logging
immutable record stored locally
9. Separation of Duties (STRICT)
The system MUST enforce:
No single individual may approve and execute a high-risk action
No experiment creator may be sole approver
Safety Reviewer must be independent for high-risk operations
10. Revocation and Emergency Control
10.1 Revocation Authority
Authorized roles MAY revoke:
experiments
plugin activations
active pedagogical changes
10.2 Revocation Effects
Upon revocation:
immediate halt of affected process
rollback to last stable state
preservation of all collected data
mandatory audit event creation
10.3 Emergency Stop (GLOBAL OVERRIDE)
System MUST support:
instant termination of all experimental or adaptive behavior
revert to baseline deterministic tutoring mode
Accessible only to:
System Administrator
Safety Reviewer (in emergency mode)
11. Audit and Traceability (MANDATORY)
Every authorized action MUST generate an immutable audit record:
{
  "event_id": "AUD-001",
  "authorization_id": "AUTH-001",
  "action_type": "experiment_activation",
  "timestamp": "2026-04-17T12:06:00Z",

  "pre_state_hash": "abc123",
  "post_state_hash": "def456",

  "result": "SUCCESS"
}
12. Offline and Low-Connectivity Mode
The system MUST support:
offline authorization verification
delayed synchronization of approvals
queued validation upon reconnection
Rules:
NO action executes without local signature validation
remote validation is supplementary, not primary
13. Prohibited Actions (ABSOLUTE)
The following are strictly forbidden:
LLM-generated self-approval of changes
implicit or undocumented authorization
retroactive approval after execution
bypass via debug flags or test modes
unsigned experiment activation
silent modification of governance rules
14. Relationship to Other Systems
14.1 Overrides
This specification OVERRIDES:
EXPERIMENTATION_FRAMEWORK (for execution authorization)
RUNTIME_COMPOSITION_ENGINE (for execution gating)
ADAPTATION_POLICY_ENGINE (for behavioral changes)
14.2 Dependency Model
Experimentation Framework → requests authorization
Composition Engine → checks authorization before execution
Adaptation Engine → cannot modify policies without approval
15. Design Rationale (Non-Normative)
This system exists to ensure:
1. Accountability
Every behavioral change is traceable to a human decision.
2. Safety
No uncontrolled optimization loops.
3. Scientific integrity
Experiments remain bounded and reviewable.
4. Resistance to drift
Prevents gradual ideological or behavioral bias accumulation.
16. Acceptance Criteria
A compliant implementation MUST:
enforce authorization before execution of governed actions
validate all cryptographic signatures
reject unauthorized or incomplete approvals
maintain immutable audit logs
support offline validation mode
enforce separation of duties
